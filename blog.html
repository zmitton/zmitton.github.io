Databases April 24th, 2014

SQL Injection

SQL injection is a problem that exist when user input is being mixed with a database statement. This situation as a whole is basically unavoidable; Data from users will always have a roll in the queries made by a backend server.

For example, lets say I have a website with user accounts. I inevitably will have a box for someone to give me their username. Then I would most certainly take that input and use it to pull up their account. Here is some ruby that could potentially create an SQL query statement:

# input_from_user = "cool_user_1"
sql_text = "select * from users where username like '#{input_from_user}';"
# sql_text = "select * from users where username like 'cool_user_1';"

Assuming the users username is "cool_user_1", and they type the into the box, the variable sql_text will be straightforward, the program will work as expected, and the account will be returned.

However, if the user wants to break into the system they could wright something like this into the input box: %; -- then the sql_text variable would become:

select * from users where username like '%; --';

the sql symbol -- is a comment, so everything after it means nothing. Also the symbol % is a "wildcard" operator. It matches with anything. Affectively this statement will return every user account. The hacker has "broken in" to the database using SQL injection.

I assume most actual implementations are a bit more complex and clever, but the principle is essentially the same.

Parameterized queries

There are a few ways to avoid this, but the best by far is by implementing parameterized queries. I am under the opinion that adding, fixes, and aversions sprinkled in here and there is a crap way to look at security. Some people will fully working systems may use these techniques in order to save money, but making all the queries parameterized virtually eliminates the risk of sequel injection.

Parameterized queries are a special way of making a query where you keep the statement and the input separated. Here is an example of the same situation using this technique:

# input_from_user = "cool_user_1"
sql_text = "select * from users where username like :input_from_user;"
sql_method(sql_text, input_from_user)

Now the example above is more like "suedocode", but the basic idea is that the user input and the rest of the statement is kept separate, and the database query uses a method that knows this. The method knows that the second argument is only to be interpreted as a matching pattern, and not as a command.

All the programmer needs to worry about is sending the command off in this way, and the rest is done behind the scenes.

NoSQL

More descriptively non-relational databases is a term given to data storage systems that don't follow the standard convention of having entries with fields and tables. Obviously It is possible to store data in another way, and any other way can be dubbed "NoSQL". The purpose of doing such a thing, usually is for performance optimization. This can be optimal when small performance gains are desired, over readability and ease of use. Usually big companies with huge server warehouses are concerned with these performance increases. After all they are paying for servers in terms of electricity. For small applications, the more organized, and easier to understand SQL relational databases are sufficient.

General Optimization

Other ways to optimize a database include making normalizing and indexing:

Normalization

Normalizing a database is basically all about not repeating data. This does two things: It help keep the database smaller, and it prevents against data conflicts. The practice of doing this involves a lot of linking. If a customer has an address, and they create an order. The order entry saved should include a field for address_id, not the whole address just copied into this other table. You already have that data, so you should just link to it. Later when you need to pull up data that is normalized, it may take a lot of "join" commands. This is a drawback, but generally is considered worth it. Data conflicts can arise when you store the same data in two different places. Lets say eventually, somewhere in your app you add a feature to change the address. Its extremely difficult to remember that the address has now been stored in an address table entry, and in every order entry by that customer. when you don't save redundant information in your database, this is imposible to happen.

Indexing

When you create a field in an SQL database you can choose at any time for it to be an "index". Basically this makes it faster to search though when you are looking up entries using WHEN that_field = "some_keyword". What the database does, is create a hash-table for that field. The database can find the entry using binary sort methods that are much faster than the usual method of looking through the database one entry at a time from start to finish. This can also have a negative affect if you are inserting entries into the database more often then searching. Every time you insert a new entry into the table, the hash-table has to be re-organized. Generally something like "user_id" should probably be an index because it is used to relate everything back to a user.

Functional vs. Imperative Programming April 17th, 2014

One of the newest programming paradigms to be talked about is "functional programming". Depending on how strict you want to get with it, its a style that uses no variables, and no "states". It achieves tasks using only functions. Because functions only depend on input and output, and functional languages don't have states, the behavior of a program can be much more predictable. With other styles of programming, it can be virtually imposible to test every possible situation for unusual behavior. There are some languages that allow for this style, they must have "first class" functions. This means that functions can be passed as arguments just the way that variables usually are. Javascript and Ruby are two languages that do this. Languages Like "C" do not. Other languages are strictly functional. These languages dont even allow for conventional variables. One very popular strictly functional language is called Haskell. Many believe this style of programming may be the future of large software projects.

Uses

Strictly functional languages right now are very talked about in academia, but are seldom used in business applications. I believe this is because they are harder to understand. I have read a lot about them and still don't really understood how to get around using variables. On the other hand. Languages with blocks and lambdas can use functional techniques when appropriate. Many of these languages are very popular and widely used.

Outlook

Although they have many advantages, the fact of the matter is, the world will not begin to use strictly functional languages anytime soon. For the most part, companies don't have the budget to redo all of there code. Even new systems will most likely use a mix of both functional and imperative styles, because many of the differences with are both advantages and disadvantages. Sometimes functional implementations are more efficient; Sometimes less. Sometimes they are easier to code; Other times harder. Sometimes they are easier to read; Other times much more difficult. Whether or not a company decides to implement functional languages in the future depends on a number of things including how well the concepts happen to lend themselves to the specific application. Some uses of functional concepts are more straightforward, while others are not. One thing is for sure: There will be some transitioning in the near future toward many of these useful techniques

The Stereotype Threat April 14th, 2014

Claude M. Steele is a Social Psychologist who has spent most of his career researching and experimenting with a phenomenon he calls the stereotype threat. He tries to understand social interactions in everyday life. He started his research wondering why ethnicities, and groups with negative academic stereotypes seem to actually underperform in schools. His goals include finding ways to integrate races more harmoniously in ways that everyone can flourish independent of race and negative stereotypes.

Background: What is it?

The stereotype threat is a phenomenon where stereotypes, especially negative ones, actually affect cognitive outcomes. Countless controlled experiments have confirmed this. Basically, if you remind someone of the negative stereotype before taking a test, they will perform significantly worse ( 0.6 standard deviations was a notable amplitude of this affect). The experiments themselves were quite convincing. Mixing around opposing stereotypes for asian women for instance, you can cause them to perform better or worse on a math test, depending on the stereotype you focus on. This work seems to prove its own significance. Obviously when someones identity affects their work, it becomes an important thing to consider.

Steele suggested the reasons for people underperforming under the weight of a negative a stereotype may go like this: When the student gets to a tough question, they begin fall back on the stereotype. As much as they want to disprove it, they become frustrated, and their mind begins to race. There is physical evidence documenting the persons nervous system becoming heightened as well.

My Experience with The Stereotype Threat

We all have personal identities, and the ones that affect us the most, are the ones that are brought to our attention most often in social situations. For instance I'm Italian. I'm proud of that, and I identify myself as such, but you can't tell by looking at me, and it doesn't really come up much. For me, this identity isn't affecting me too much in the ways relevant to this work. I also have never really heard of many limiting stereotypes attached to being Italian.

Steele has done experiments where he told a white student he/she was going to have an open discussion about affirmative action. He showed the person pictures of who they would be talking to. In some cases the pictures were of black people, and in others, they were white people. The idea of having a conversation about race with black people, scared them. They would arrange the chairs in the room much further appart. This is one thing I can relate to. Nobody wants to be perceived as racist, myself being a white male that considers himself not to be racist, can relate to their nervousness. A white male might have a "lot to lose" in that situation. It could affect my life negatively to be perceived as a racist, and therefore I sometimes try to avoid that type of confrontation altogether.

My Plan: How to Break the Cycle

Recognizing these stereotypes and realizing they are only as true as you believe they are, can help immensely. Looking past them and working together can have a positive affect as well. Steele talked about his own experience in grad school with an advisor that took him under his wing, and treated him like anybody else. He believes that relationship had a huge positive impact on him. Race was an identity that did not affect their relationship, and the mentor confirmed regularly simply through his actions, that Steele had as much to contribute as anyone else.

Steele discussed an experiment on how to give good feedback as well. In situation where race barriers exist is is especially difficult to have well accepted feedback. The results of some of his feedback studies were very surprising as well. The best way to give feedback, according to his research, may have little to do with stereotypes or race. Letting the person know that you believe in their ability to do this particular type of work is the most productive way to give feedback. Telling them you think the world of them "in general" did not help at all, in Steele's studies.

Object Properties and Methods: When to use public and private settings April 10th, 2014

Object Methods

You can set them as private, or by default they will be public

First off, what does making a an class method private even do? Normally, a method defined in a class is available to any object instance of that class. If you make the method private, the instance object can only call the method from within the class definition.

In the following example, the print_board method is private. It is being called from this game_over function which is inside the class definition.

class Battle_game
   def game_over (outcome)
     puts "You #{outcome}.\n\n"
     print_board(@computer_board)
   end

   private
   def print_board(the_board)
     the_board.each_with_index{|x, i|
       x.each{|y| print y }
     }
   end
end

When to use this?

In the battleship game example, let say you are defining the class, but other programmers are defining different pieces of the game. You want to make a method that prints the board print_board() (showing where all the ships were), to display after someone wins the game. You dont want another programmer to accidentally use this in a way that reveals it during the game. You can define that method as private and only call it from the game_over() method. (a method only called when the game ends).

You really might want to make most of the methods private, and only have a few public ones like take_turn() and display_game_status().

Object properties: When to set them as readable/ writable, and when not to

As with methods, an object's properties have accessibility settings as well. Within a class definition creating object properties is as simple as defining them using an @ symbol. For example @num_misses_so_far. In ruby, by default, you wont be able to examine or manipulate this property from outside the walls of your class definition. Analogous to having the "private" setting we just talked about. Only with properties, they are private by default. If you want to be able to read the property or write to it from the outside, you need to add an attr statement:

class Battle_game
   attr_reader :num_misses_so_far # property is publicly readable
   attr_writer :num_misses_so_far # is publicly writable
   attr_accessor :num_misses_so_far # is publicly readable and writable
end

In practice, you would only ever use one of the three statements above, depending on what kind of access you want. Now you can access this property from anywhere if you have a battle_game_instance: battle_game_instance.num_misses_so_far.

Class Properties

You can also store variables pertaining to the class itself. They are not specific to an individual instance, and they are often used to store class settings like how many guesses the Battle_game allows @@allowed_guesses = 10. These variables are always defined within the class using two @ symbols in a row: @@. Again, by default, the variable cannot be seen outside of the class definition. To access this variable you must define within the class a "getter" (or "setter"): A method usually of the same name that simply returns the class variable (or sets it). An example of a possible class variable and its getter function for our game might be

class Battle_game
   @@allowed_guesses = 10

   def self.allowed_guesses
     @@allowed_guesses
   end

end

Accessing this now is done by calling the allowed_guesses method directly on the class itself (not an instance of it) like so: Battle_game.allowed_guesses.

Ruby Conventions

There are other languages that just make all these variables and others public by default making development a lot more rapid. There are pros and cons to each way of doing things. Remembering all of this is a bit confusing, but I think it makes programs faster, and less breakable (2 qualities that can be very important when working with large teams, on immense projects).

My Battleship Game

In an effort to properly use these concepts of private vs public object variables and methods, I decided to make this "battleship like" game. I call it "battleship like" because starting with the game of battleship, I changed the following rules, to make my example program a bit simpler

  1. Single-player game. you just make guesses against the computer
  2. Computer doesn't take any turns
  3. You win by sinking all the computer's ships in <= 10 turns
  4. All ships take up just 1 spot

Take a look at the variables, and methods and try to understand why I gave them the privacy settings (or lack thereof) that I did.

Here is a link to the working program on github: Battleship.rb

class Battle_game

   attr_reader :num_misses_so_far , :game_in_progress , :guessing_board # readable properties
   attr_accessor :title # this is the only instance property that can be read and written to
  
   # The following variables belong to the class. Here they control game settings
   @@allowed_guesses = 10
   @@board_length = 4

     @title = game_title

     i = (@@board_length ) - i.to_i

       @computer_board[i][j] = "X"

       @num_misses_so_far += 1

       "Miss"

end

Enumerators April 3rd, 2014

Some of the most useful functions in ruby are enumerators. They allow the programmer to loop through an array, hash, or numerical set, while and indicate a block of to perform on it. There are many of these enumerator functions, one of the easiest to understand is the .map{|item| block } function.

map and map!

The map function performs the block once for each element in the group calling it. It creates a new array of all the returned values. If the "!" is indicated, the original array itself can be modified.

Examples:

array1 = [1,2,3,4,5]
array2 = array1.map{|x|
     x*10
}
puts array1 # [1, 2, 3, 4, 5]
puts array2 # [10, 20, 30, 40, 50]

In the above code the map function multiplies each element of array1 by 10 and outputs them as one new array which we assign to "array2". As you can see from the output array1 remains unchanged

If the exclamation point is used the original array can itself be modified instead:

array1.map!{|x| # notice the exclamation point this time
     x*10
}
puts array1 # [10, 20, 30, 40, 50]

This time the original array "array1" is modified, so there is usually no need for a second array to store the results

Although the results of this looping method may seem trivial I believe that's sort of the point. Doing this without using .map{} would take a few extra lines of code and would require a few temporary variables. Similar structures are made over and over by programers. Because of their simplicity, they tend to come up a lot. Having a built in method saves time for the coder, and the reader, while making the code smaller, slightly more efficient, and more elegant.

Why is Javascript so Popular March 20th, 2014

Don't let the name confuse you.

Javascript is not the same languages as Java. Javascript has almost no connection to Java at all. They have some similar syntax, but are very different types of languages, and work very differently. Java must be compiled before it can be used, while Javascript can be interpreted by a computer as-is because it is actually compiled live (just before it is executed). It actually used to be called "Livescript" for this reason. The suffix "script" can be misleading as well. Many times a language is considered "scripting" when it is specialized for just certain types of tasks. Javascript is actually a very powerful full language.

Javascript is the language of the World Wide Web.

The internet is a big place with many different parts. Most people use a browser program to view web pages on the World Wide Web through there internet connection.
There are many other ways to communicate with this big connected network: For instance when you use an app on your phone that receives data. The connection is still needed for your phone to pull up the data, but this is not considered the "web", because you are not using a browser program.
Businesses often have desktop applications that use another type internet connection for their inventory systems. Inventory across multiple stores can be kept track of on one "cloud" server. That way each store has data that instantly reflects any changes that occur from any other stores.
As you can see, there are many different ways to comunicate with the internet but, by far, the most popular one is through a browser, on the web, and when it comes to browsers: All major browsers use Javascript. That means, that if a developer wants his website to work correctly for everybody, they pretty much have to use javascript.

Javascript is a Client Side Language

Many functions of a web page need to happen fast. The problem is: any action that requests data from the internet takes at least about a second. Thats usually not good enough. Enter Javascript. Javascript programs run on the user's (or client's) computer. When Javascript runs, it does not have to talk to the internet, it all can take place within the users computer, making it much much faster. When you click a button in javascript, it can take less than 1/1000 seconds: virtually instantaneously. Javascript is the reason everything you click within a web page isn't painstakingly slow.

Javascript can do almost anything on a web page

In modern browsers the content of a site is loaded into javascript objects occurring to a convention called the DOM (document object module). Because of this, every single thing within a web page is available to javascript. Therefore developers can manipulate anything on the page, and write code in javascript to do that.

Favorite Sites March 13th, 2014

eBay

This has always been one of my favorite sites, its simple to understand and its always really easy to use. When you load the page, it forms perfectly to your current screen size no matter how big that may be. Right in fron, are tons of items I've viewed, or purchased in the past. This can seem a bit cluttered, but they are wnly pictures of other products, so the extra content does not add any confusion. This really is where the page shines, because the internet is a huge place, and the functionality of being able to narrow down what you want in an affective way makes all the difference.
